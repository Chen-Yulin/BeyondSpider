# ADR 0013: 雷达按通道筛选可跟踪物体，筛选同时决定显示与可锁定范围

日期：2026-07-17
状态：已实现，待游戏内验证

## 背景

3D 雷达图（`CaptainRadarView`）把 `ship.Tracks` 里所有 `TrackKind.Ship`/`TrackKind.HeavyMissile`
无差别画出来。一场规模稍大的交战里，几十枚小导弹会把舰船光点淹没，玩家既看不清战场，也没法在通道
1~3 上点中想打的那艘船。同时四条火力通道（ADR-0010）共用同一份显示，通道之间无法各自关心不同的
目标类型。

用户要求（四条，本 ADR 只记录其中有架构含义的第三条，其余三条见"其他一并实现的改动"）：雷达顶部
新增筛选栏，可筛选**舰船 / 重型导弹 / 中导弹 / 小导弹**；只显示勾选的类型；**每个通道可以有不同
配置**；**只能锁定筛选显示的物体，0 通道的自动防空通道也是如此**。

## 决策

1. **新增 `RadarFilter.cs`**：`RadarCategory` 四行枚举（Ship/HeavyMissile/MediumMissile/
   SmallMissile）+ 位掩码工具 + `TryGetCategory(ITrackable)` 分类。分类不能只看 `TrackKind`——
   三种尺寸的导弹**全部**上报 `TrackKind.HeavyMissile`，S/M/L 只能靠物理 `Radius`（
   `MissileLauncherAssets.MissileRadius` = 3/5/10）区分，阈值取三者中点（4 / 7.5）。这套分级原本
   以私有常量的形式重复实现在 `CaptainRadarView.MarkerScaleFor`（决定光点大小）里，现在收敛成
   `RadarFilter` 一处——筛选栏的行和光点的尺寸档**必须**是同一套判定，否则会出现"看得见但筛不掉"
   的类型。

2. **掩码存在 `ShipState.ChannelFilters[4]`**（每通道一个 `RadarFilter` 位掩码，默认全开
   `AllMask`），不是存在雷达视图里。因为消费方有两个：客户端本地的雷达图，和**主机侧**的通道 0
   自动防空（`SpaceCaptainBlock.UpdateAutoAirDefenseLock` 跑在 `SimulateFixedUpdateHost`）。存在
   视图里的话，客户端改筛选对主机的自动选目标不会有任何影响。

3. **新增 `CaptainLockNet.FilterMsg`**（shipPlayerId, shipCoreGuidHash, channel, categoryMask）
   同步筛选。沿用 `LockMsg` 的既有约定：雷达图是客户端本地的，改动走 `ModNetworking.SendToAll`
   （`TryLockAtMouse` 早就是这么做的），主机收到后写进 `ShipState`。筛选影响通道 0 的自动锁定，
   属于"能影响战斗结果的状态"，按 mod 指南必须同步而不能只留在本地。

4. **显示与锁定用同一份筛选，靠构造保证一致**：`SyncMarkers` 用**当前激活 tab** 那条通道的掩码
   决定画哪些光点；点击锁定（`FindLockableTarget`）**只从已画出的光点里选**，锁的又正是那条激活
   通道。所以"只能锁定筛选显示的物体"不是一条额外校验，而是数据流本身的结果——不存在"筛选放过了
   显示但漏了锁定"的窗口。

5. **筛选收紧时，主机侧掉锁**：`SpaceCaptainBlock.PruneChannelTargets`（原
   `PruneDeadChannelTargets`）除了清理已死目标，现在也清理"本通道筛选不再允许"的锁——否则玩家取消
   勾选后，一个**看不见的锁**仍在继续指挥武器开火。掉锁复用既有的解锁广播，通道 0 的手动锁掉了会
   交还给自动选择。

6. **裸炮弹（`TrackKind.LargeProjectile`）不受筛选约束**：筛选栏只有用户明确列出的四行，炮弹既没有
   对应行、雷达也从来不给它画光点（ADR-0012 已记录这一点）。如果让它走同一条 `Allows` 判定，它会
   **永远匹配不上任何一行**，结果是通道 0 对炮弹的点防御被静默且永久地关掉，而 UI 上没有任何开关能
   把它打开。因此 `RadarFilter.Allows()` 对"分不出类别"的目标一律放行，`RadarFilter.Shows()`（管
   显示）则一律不画。这是本 ADR 唯一偏离"只能锁定筛选显示的物体"字面表述的地方，是为了不引入一条
   玩家无法察觉、也无法恢复的防御能力回退。

   如果以后希望炮弹也能被筛掉，正确做法是给筛选栏**加第五行**（并让雷达真的给炮弹画光点），而不是
   让它落进现在这条放行分支。

## 其他一并实现的改动（无架构分歧，记录以便回溯）

- **雷达扫描 0.25s → 0.05s（4Hz → 20Hz）**，`RadarPanelBlock`。顺带把 `nextScanTime` 从
  "now + interval" 改成对齐到全局栅格 `(floor(t/Δ)+1)·Δ`：一艘舰的多个雷达面必须在**同一个
  FixedUpdate** 扫描，`ship.Tracks` 的"先清空再追加"合并才不会互相清掉对方的轨迹。原来 0.25s 间隔
  配 0.05s 合并窗口只是碰巧成立（所有雷达同帧开始模拟、天然同相），间隔压到 0.05s 后窗口跟着收窄，
  模拟中途补放置的雷达面会以自己的相位扫描并每轮清掉同伴的轨迹。栅格对齐把这个巧合变成了保证。
- **光点平滑插值**：`SensorTrack` 新增 `ScanTime`（CONTEXT.md 里 传感器轨迹 的定义本来就含"最后
  更新时间"，只是结构体一直没有）。`CaptainRadarView` 每个光点按 `Position + Velocity × 轨迹年龄`
  推算，再指数逼近该推算值。**必须是"先按速度推进、再逼近"而不是单纯 lerp**：单纯 lerp 会让光点稳定
  落后真实位置 `速度/平滑率`（500m/s 的导弹在 12/s 的平滑率下要落后 ~40m），推进项负责跟上运动，逼近
  项只用来吸收每次扫描带来的那一小步误差。平滑状态存**世界坐标**而非显示坐标，这样本舰转向和缩放
  始终精确、即时，只有目标自身的运动被平滑。
- **点击改成"就近抓取"**：原来靠给每个光点挂 `SphereCollider` + `Physics.RaycastAll` 精确命中。小
  导弹的光点世界半径只有 `0.3 × 0.3 × 0.25 ≈ 0.02` 单位（显示半径是 4），实际上点不中。改成把每个
  已画出的光点用 `radarCamera.WorldToScreenPoint` 投影到雷达纹理像素空间，取离点击点最近且在
  `ClickGrabRadiusPixels`（30 纹理像素）以内的可锁定光点。"最近优先"意味着阈值放宽也不会误抓——密集
  目标仍然解析到真正最近的那个。副作用是口袋场景里的碰撞体、`markerToTrackable` 反查表、
  `TrackableFor()` 全部不再需要，一并删除。
- **垂线**：每个光点向雷达平面画一条白色垂线（`LineRenderer`，自带朝相机 billboard，orbit 时不会
  看没）；被任意通道锁定的目标垂线加粗、并延长出"垂足 → 雷达原点"的一段，同一条折线走完
  `光点 → 垂足 → 原点`。本舰光点在原点，没有垂线可画。

## 后果

- 玩家可以让通道 0 只管小导弹（点防御）、通道 1 只管舰船（主炮），互不干扰——这是四条通道第一次真正
  能各自关心不同的战场切片。
- 筛选是 opt-out：默认全开，没碰过筛选栏的存档/玩家行为与改动前一致。
- `ShipState.ChannelFilters` 不持久化到存档（`ShipState` 每场模拟由分区重新生成），每场模拟从全开
  开始。如果以后玩家反馈"每次开局都要重设筛选"，应该做成舰长上的一个 mapper 字段，而不是给
  `ShipState` 加存档。
- 通道 0 的筛选和 IFF 是两道**独立**的门（先 IFF、后筛选），语义不重叠：IFF 管"这是不是敌人"，筛选
  管"我这条通道关不关心这种东西"。
- 雷达扫描频率提高 5 倍后，`ship.Tracks` 的刷新率也成了火控提前量的刷新率，理论上对高速横向目标的
  命中率会有可感知的提升——这一点尚未在游戏内量化验证。
